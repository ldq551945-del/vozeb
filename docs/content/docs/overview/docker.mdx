# Docker 部署

如果你希望在自己的机器或服务器上运行 VOZEB，推荐直接使用 Docker Compose。

## 使用发布镜像

项目根目录的 `docker-compose.yml` 已经指向 VOZEB 发布镜像：

```yaml
services:
  app:
    image: ghcr.io/csyqlz/vozeb:latest
    container_name: vozeb
    ports:
      - "${VOZEB_BIND_HOST:-0.0.0.0}:3000:3000"
    volumes:
      - vozeb-data:/app/web/.data
    environment:
      NEXT_PUBLIC_SITE_URL: ${NEXT_PUBLIC_SITE_URL:-}
      NEXT_PUBLIC_DOC_URL: ${NEXT_PUBLIC_DOC_URL:-}
      VOZEB_COOKIE_SECURE: ${VOZEB_COOKIE_SECURE:-}
      VOZEB_DATA_DIR: /app/web/.data
      VOZEB_INTERNAL_ORIGIN: http://127.0.0.1:3000
    restart: unless-stopped

volumes:
  vozeb-data:
```

部署命令：

```bash
git clone git@github.com:csyqlz/vozeb.git
cd vozeb
docker compose up -d
```

启动后访问：

```text
https://你的域名
```

默认 `docker-compose.yml` 使用发布镜像 `ghcr.io/csyqlz/vozeb:latest`，并通过 `VOZEB_BIND_HOST=0.0.0.0` 同时兼容 `http://服务器IP:3000` 直连和域名反向代理。先确认容器状态：

```bash
docker compose ps
curl -I http://127.0.0.1:3000
```

如果 IP 可以访问但域名打不开，请先把域名 A/AAAA 记录解析到服务器，再用 Nginx、Caddy 或宝塔把域名反向代理到 `http://127.0.0.1:3000`。反代需要保留 `Host`、`X-Forwarded-Host`、`X-Forwarded-Proto`，否则 HTTPS 登录 Cookie 和参考图公网临时地址可能判断错误。域名部署建议复制 `.env.example` 为 `.env` 并填写：

```env
NEXT_PUBLIC_SITE_URL=https://你的域名
VOZEB_COOKIE_SECURE=true
```

如果你只想允许域名反代访问，不想暴露 `服务器IP:3000`，在 `.env` 里追加：

```env
VOZEB_BIND_HOST=127.0.0.1
```

当前 `docker-compose.yml` 会把这些 `.env` 变量传入容器。Nginx 示例：

```nginx
location / {
    proxy_pass http://127.0.0.1:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
}
```

发布镜像版端口配置：

```yaml
services:
  app:
    image: ghcr.io/csyqlz/vozeb:latest
    container_name: vozeb
    ports:
      - "${VOZEB_BIND_HOST:-0.0.0.0}:3000:3000"
```

本地构建版端口配置：

```yaml
services:
  app:
    image: vozeb:local
    pull_policy: build
    build:
      context: .
      dockerfile: Dockerfile
    container_name: vozeb
    ports:
      - "${VOZEB_BIND_HOST:-0.0.0.0}:3000:3000"
```

默认 `VOZEB_BIND_HOST=0.0.0.0`，IP 直连和域名反代都可用。如果只想允许域名反代访问，在 `.env` 设置 `VOZEB_BIND_HOST=127.0.0.1`。

第一次注册的账号会自动成为管理员。请用这个首个管理员账号进入后台，完成网站标题、Logo、SEO、邮箱注册、SMTP、模型渠道、默认模型、用户积分和公共提示词库配置。

## GHCR 拉取 denied

如果服务器执行 `docker compose pull` 或 `docker compose up -d` 时提示类似 `denied`、`unauthorized` 或 `permission_denied`，通常有两个原因：

- GitHub Packages 中的 `ghcr.io/csyqlz/vozeb` 还没有设为 Public。
- 你想保持镜像私有，但服务器没有执行 `docker login ghcr.io`。

公开部署推荐进入 GitHub Packages 的 `vozeb` 包页面，打开 `Package settings -> Change visibility`，把可见性改为 `Public`。如果还有文档站镜像 `vozeb-docs`，也需要同样设置。

私有部署则在服务器上登录 GHCR：

```bash
echo "你的 GitHub Token" | docker login ghcr.io -u 你的GitHub用户名 --password-stdin
docker compose pull
docker compose up -d
```

Token 需要具备读取 Packages 的权限。不要把 Token 写进仓库或公开日志。

## 本地构建镜像

如果需要基于当前源码构建镜像：

```bash
docker compose -f docker-compose.local.yml up -d --build
```

0.5G 到 2G 内存的小服务器建议优先使用发布镜像；服务器只拉取并启动容器，不现场执行 `next build`。如果服务器只有 0.5G 内存，可以使用低内存 Compose 文件，它同样只拉取 `ghcr.io/csyqlz/vozeb:latest`，并给运行中的容器加上 512MB 内存限制：

```bash
docker compose -f docker-compose.lowmem.yml pull
docker compose -f docker-compose.lowmem.yml up -d
```

0.5G 服务器不要执行 `docker compose up -d --build`。需要自定义源码时，请在本机或 CI 构建后推送镜像仓库，再让服务器拉取。

## 为什么低配服务器构建会拉满 CPU

这不是异常崩溃，而是 Next.js 生产构建阶段的正常资源峰值。当前 VOZEB 已经比原作者早期版本多了账号系统、管理员后台、公共提示词种子、文档资料、standalone Docker 输出和页面生成流程；`next build` 会编译、收集页面数据并生成静态页面，所以 2 核 2G 机器上 CPU 可能长时间接近 100%/200%，内存不足时还会用到 swap。

原作者早期版本更偏纯前端个人画布工具，构建面更小，也没有现在这些后台和内置数据步骤，因此同规格服务器压力会低很多。

建议顺序：

1. 0.5G 服务器使用 `ghcr.io/csyqlz/vozeb:latest` 发布镜像，服务器只拉取和启动，适合 2 分钟级部署。
2. 必须本地构建时保留 `NEXT_BUILD_CPUS=1`，并开启 BuildKit 缓存；建议至少 2G 内存，不建议 0.5G 机器现场构建。
3. 如果仍然卡死，在本机或 CI 构建镜像后推送到镜像仓库，再让服务器拉取。

## 更新镜像

使用发布镜像部署时，更新到最新版本只需要：

```bash
docker compose pull
docker compose up -d
```

如果你看到本地仍在运行 `vozeb:0.5.1-xxxx` 一类旧镜像，说明当前容器不是从 `ghcr.io/csyqlz/vozeb:latest` 更新而来。请改用本仓库新的 `docker-compose.yml`，或重新构建本地镜像。

## 文档站镜像

文档站位于 `docs/` 目录，按带服务端能力的 Next.js standalone 应用单独构建，不打进主应用镜像。

使用发布镜像：

```bash
cd docs
docker compose up -d
```

基于当前源码本地构建：

```bash
cd docs
docker compose -f docker-compose.local.yml up -d --build
```

## 数据说明

当前主应用镜像只启动 Next.js。画布、我的素材、生成记录和 AI API Key 默认保存在浏览器本地。

登录状态通过服务端 Session Cookie 保持。使用 `http://服务器IP:3000` 直连时，VOZEB 会写入可在 HTTP 下保存的 Cookie；使用 HTTPS 反向代理时，请保留 `X-Forwarded-Proto: https`，VOZEB 会自动启用 `Secure` Cookie。若你的代理环境比较特殊，可以在容器环境变量中设置 `VOZEB_COOKIE_SECURE=true` 或 `VOZEB_COOKIE_SECURE=false` 强制指定。

账号、后台网站设置、用户积分、签到记录和公共提示词库会写入服务端 `.data` 目录：

- 用户与后台设置：`/app/web/.data/auth.json`
- 管理员公共提示词：`/app/web/.data/prompts.json`
- 生成日志：`/app/web/.data/generation-logs.json`
- 生成结果服务器副本：`/app/web/.data/generation-assets`

本仓库的 `docker-compose.yml` 和 `docker-compose.local.yml` 默认把命名卷 `vozeb-data` 挂载到 `/app/web/.data`，并通过 `VOZEB_DATA_DIR=/app/web/.data` 固定服务端数据库目录。正常执行 `docker compose pull && docker compose up -d` 更新镜像不会删除已有用户、后台设置、生成日志或服务器副本；只有执行 `docker compose down -v`、手动删除 Docker volume，或使用没有挂载 `.data` 的旧容器才可能导致数据丢失。

图片和视频结果的服务器副本是否保存，由管理员后台系统设置里的“图片/视频下载到服务器”分别控制；是否允许普通用户把这些副本作为最后兜底访问，则由“图片/视频服务器兜底”分别控制。关闭下载保存不会删除旧副本，旧副本可以在后台生成资源统计中清理。
