# Docker 部署

如果你希望在自己的机器或服务器上运行项目，可以直接使用 Docker Compose。

## 使用发布镜像

项目根目录的 `docker-compose.yml` 已经指向 VOZEB 发布镜像：

```yaml
services:
  app:
    image: ghcr.io/csyqlz/vozeb:latest
    volumes:
      - vozeb-data:/app/web/.data

volumes:
  vozeb-data:
```

```bash
git clone git@github.com:csyqlz/vozeb.git
cd vozeb
docker compose up -d
```

启动后访问：

```text
http://localhost:3000
```

## 本地构建镜像

如果需要基于当前源码构建镜像：

```bash
docker compose -f docker-compose.local.yml up -d --build
```

0.5G 到 2G 内存的小服务器建议优先使用发布镜像；服务器只拉取并启动容器，不现场执行 `next build`。如果必须在 0.5G 服务器上尝试本地构建，请先开启 1G 到 2G swap，并使用低内存构建配置：

```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
docker compose -f docker-compose.lowmem.yml up -d --build
```

低内存构建会把 Node 构建内存上限压到 384MB，并保留 `NEXT_BUILD_CPUS=1`，能降低 OOM 概率，但会明显变慢。如果仍然失败，请改用发布镜像，或在本机/CI 构建后推送镜像。

## 为什么低配服务器构建会拉满 CPU

这不是异常崩溃，而是 Next.js 生产构建阶段的正常资源峰值。当前 VOZEB 已经比原作者早期版本多了账号系统、管理员后台、公共提示词种子、文档资料、standalone Docker 输出和 Turbopack 页面生成流程；`next build` 会编译、收集页面数据并生成静态页面，所以 2 核 2G 机器上 CPU 可能长时间接近 100%/200%，内存不足时还会用到 swap。

原作者早期版本更偏纯前端个人画布工具，构建面更小，也没有现在这些后台和内置数据步骤，因此同规格服务器压力会低很多。

建议顺序：

1. 0.5G 服务器使用 `ghcr.io/csyqlz/vozeb:latest` 发布镜像，服务器只拉取和启动，不现场构建，才适合 2 分钟级部署。
2. 必须本地构建时保留 `NEXT_BUILD_CPUS=1`，并开启 BuildKit 缓存；建议至少 2G 内存。
3. 如果仍然卡死，在本机或 CI 构建镜像后推送到镜像仓库，再让服务器拉取。

## 更新镜像

使用发布镜像部署时，更新到最新版本只需要：

```bash
docker compose pull
docker compose up -d
```

如果你看到本地仍在运行 `vozeb:0.5.1-xxxx` 一类旧镜像，说明当前容器不是从 `ghcr.io/csyqlz/vozeb:latest` 更新而来。请改用本仓库新的 `docker-compose.yml`，或重新构建本地镜像。

## 文档站镜像

文档站位于 `docs/` 目录，按带服务端能力的 Next.js standalone 应用单独构建，不打进主应用镜像。

使用发布镜像：

```bash
cd docs
docker compose up -d
```

基于当前源码本地构建：

```bash
cd docs
docker compose -f docker-compose.local.yml up -d --build
```

## 数据说明

当前主应用镜像只启动 Next.js。画布、我的素材、生成记录和 AI API Key 默认保存在浏览器本地。

账号、后台网站设置、用户额度、签到记录和公共提示词库会写入服务端 `.data` 目录：

- 用户与后台设置：`/app/web/.data/auth.json`
- 管理员公共提示词：`/app/web/.data/prompts.json`

本仓库的 `docker-compose.yml` 和 `docker-compose.local.yml` 默认把命名卷 `vozeb-data` 挂载到 `/app/web/.data`。正常执行 `docker compose pull && docker compose up -d` 更新镜像不会删除已有用户；只有执行 `docker compose down -v`、手动删除 Docker volume，或使用没有挂载 `.data` 的旧容器才可能导致数据丢失。
